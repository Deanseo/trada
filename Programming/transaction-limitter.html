2. What's Transaction Limiter?

A Transaction Limiter helps ensure you don't accidentally send more orders than you're allowed to.

3. Why do we need Transaction Limiter?

To maintain a fair trading environment and protect individual traders, global exchanges often limit the total number of order requests a market participant can send, within a fixed time frame. Breaking this rule is never a good idea, because there will be consequences(punishment).

For example, the first offense might result in a trading suspension for a few hours. If the behavior continues, you could face a longer ban - three months, six months, or even permanently.

Don't get me wrong. These rules are essential. It's a good idea to ban those who can't keep their trading system under control. This is for everyone and to maintain fairness in the market.

3.1. Crypto Trading and Transaction Limiter

Crypto exchanges often impose even stricter restrictions. Since the fall of FTX, which supported the FIX protocol, most users trade, using the exchange's WebAPIs where the definition of the transaction limit is somewhat vague such that they're advised to make order requests in a more conservative stance.

4. How to develop Transaction Limiter?

In my opinion, it's an interesting project to write a limiter class, not only in the trading industry but in general. The implementation can be generic to your business requirements, such that you can control certain tasks to be executed up to certain times only.

For simplicity, let's say we don't support any rich features but focus on these abolustely minimal requirements:

  • Check, if we have capacity.
  • Decrement capacity by 1, if we do.

We've got those two interfaces defined, check() and decrement(), but how are we supposed to define and checking the capacity? Though there are different ways to implement that, I'll use a circular buffer in this article.

One more thing - imagine a scenario where we must send orders, no matter what, even if the limitter says no. For example, we might have to send cancellation orders due to a sudden price move or our hedging position becoming neutral.

To get around that situation, we'll also create another exclusive buffer that only accepts when the order comes with a priority tag.

4.1. Implement a base class

A base class defines those two interfaces as well as the priority tag. Note that I made the base class let the priority tag optional, rather than enforcing the priority tag to be always present but rather 

enum class priority_tag { Tag_ };

template <typename ImplType>
struct base_limitter {
    auto *This() { return static_cast<ImplType *>(this); }
    auto *This() const { return static_cast<const ImplType *>(this); }

    [[nodiscard]] constexpr auto size() const noexcept { return This()->size(); }

    bool check_capacity_and_add() { return This()->check_capacity_and_add(); }
    bool check_capacity_and_add(priority_tag) {
        if constexpr (requires(ImplType impl, priority_tag tag) {
                          { impl.check_capacity_and_add(tag) } -> std::same_as<bool>;
                      }) {
            return This()->check_capacity_and_add(priority_tag{});
        } else {
            return This()->check_capacity_and_add();
        }
    }
};
